\section{Introduction}
The rapid advancement of machine learning has driven an increasing demand for efficient hardware acceleration of neural networks. 
While general-purpose processors such as CPUs and GPUs have traditionally been used for this purpose, significant performance and energy efficiency gains have emerged from the use of specialised hardware accelerators.
In particular, field-programmable gate arrays (FPGAs) have gained attention due to their ability to provide flexible, fine-grained parallelism and custom dataflow architectures tailored to specific workloads.

A key technique enabling these improvements is quantisation, which reduces the numerical precision of neural network parameters and activations. 
By representing values with fewer bits, ranging from reduced fixed-point precision to fully binarized representations, quantisation significantly lowers memory usage and computational complexity. 
This, in turn, allows for faster inference, reduced power consumption, and more efficient utilisation of hardware resources. 
In extreme cases such as binarized neural networks (BNNs), arithmetic operations can be simplified to bitwise operations, making them especially well-suited for FPGA implementations.

Much of the recent progress in the AI hardware acceleration space can be attributed to the combination of increased hardware specialisation and aggressive quantisation strategies. 
Specialised architectures are able to exploit the reduced precision of quantised models to maximise parallelism and throughput, achieving performance levels that would be difficult to attain with conventional high precision designs.

This project explores the role of quantisation in FPGA-accelerated neural networks, with a particular focus on binarized neural networks.
Using the FINN framework, quantized models are mapped to dataflow-style hardware architectures on an FPGA platform. 
The study investigates how neural networks trained in Brevitas are mapped to FPGA-accelerated neural networks and how different levels of quantisation that impact performance, as well as how these effects interact with architectural parallelism. 
Through these experiments, the project aims to provide insight into the trade-offs between accuracy, resource utilisation, and performance in quantised neural network deployment on reconfigurable hardware.

\subsection{Training a BNN using Brevitas}
Training a Binarized Neural Network (BNN) using the Brevitas library follows a very similar workflow to training a conventional Convolutional Neural Network (CNN) in PyTorch. The overall structure of defining a model, specifying a loss function, and performing optimisation through backpropagation remains unchanged. 

The key difference lies in the use of quantised layers provided by Brevitas. Instead of standard PyTorch layers, such as \texttt{Conv2d} or \texttt{Linear}, Brevitas introduces quantised counterparts (e.g. \texttt{QuantConv2d}, \texttt{QuantLinear}, and \texttt{QuantIdentity}) that simulate reduced numerical precision during training. 
These layers allow both weights and activations to be constrained to a fixed bit-width, enabling the network to approximate the behaviour of low-precision hardware implementations.

In addition, quantisation-aware training requires careful handling of value ranges and scaling factors. For example, inputs are typically normalised to a fixed range (e.g. $[-1, 1]$), and intermediate activations are passed through explicit quantisation layers. 
Batch normalisation is essential after each layer to stabilise training, as quantized networks are generally more sensitive to changes in data distribution.

The model architecture consists of a sequence of quantised convolutional layers for feature extraction, followed by fully connected layers for classification.

Next we convert the model from QONNX to FINN-ONNX, operations provided by the FINN library only work on this intermediate representation of models:

We also perform some basic transformations to prepare the graph for upcoming transformations, these include:
\begin{itemize}
    \item \verb+GiveUniqueNodeNames+
    \item \verb+GiveReadableTensorNames+
    \item \verb+InferShapes+
    \item \verb+InferDataTypes+
    \item \verb+FoldConstants+
    \item \verb+RemoveStaticGraphInputs+
\end{itemize}
The transformations \verb+GiveUniqueNodeNames+ and \verb+GiveReadableTensorNames+ give the nodes in the graph unique names by their enumeration, tensors are then given human-readable names based on the node names.
The \verb+InferShapes+ and \verb+InferDataTypes+ transformations derive shapes and data types of the tensors from the model properties and sets them in the \verb+ValueInfo+ property of the model.
These four transformations do not affect the structure of the graph.

The remaining two transformations modify the structure of the graph, \verb+FoldConstants+ identifies nodes with constant inputs and calculates the output, then for the following node the inputs are set as constant-only inputs and the previous node is deleted.
\verb+FoldConstants+ is useful for FPGA synthesis as it reduces the number of nodes (blocks) that need to be stitched together in the final design.
The other mutating basic transformation is \verb+RemoveStaticGraphInputs+, this transformation removes any top-level graph inputs that already are associated with ONNX initialisers.
\begin{jupytercell}
qonnx_cleanup(export_onnx_path, out_file=export_onnx_path)
model = ModelWrapper(export_onnx_path)
model = model.transform(ConvertQONNXtoFINN())
model = model.transform(InferShapes())
model = model.transform(FoldConstants())
model = model.transform(GiveUniqueNodeNames())
model = model.transform(GiveReadableTensorNames())
model = model.transform(InferDataTypes())
model = model.transform(RemoveStaticGraphInputs())
model.save(build_dir + f"/end2end_{model_name}_tidy.onnx")
showInNetron(build_dir+f"/end2end_{model_name}_tidy.onnx")
\end{jupytercell}

We can see that the CNN has been transformed into a sequential network as a result of the mutating changes removing constants and static graph inputs.
For example, the first \verb+BipolarQuant+ node in Fig~\ref{fig:InitialGraph} has been moved into the \verb+conv+ node in \ref{fig:AfterTidyUp}.
From this netron representation, it is clear now that the model consists of an initial \verb+MultiThreshold+ layer which quantizes float input into 8-bit, followed by repeating convolution-convolution-maxpool layer pattern with 3x3 filters for feature extraction. Finally followed by fully-connected layers for classification.
Transforming the network into a sequential set of nodes is useful for FPGA deployment as it reduces the amount of hardware resources used to implement the network.
Additionally it allows for easier implementation of selective parallelism, which is explored more in \ref{Experiment3}.

Next we add some preprocessing and postprocessing steps to the model.
For preprocessing we want to divide the 8-bit input data by 255 so that the inputs to the network are bounded between 0 and 1.
This is achieved by creating a QONNX node using the \verb+ToTensor()+ function, converting it to FINN-ONNX and merging the models together.

For postprocessing we want to take the output of the network and return the index of of the image category with the highest probability, this should return an index from 0-9 as the model is based on CIFAR-10.
This node can be directly inserted at the end of the model via a transformation with \verb+InsertTopK()+ without having to do the intermediate representation transformations as performed in pre-processing.
\begin{jupytercell}
from finn.util.pytorch import ToTensor
from qonnx.transformation.merge_onnx_models import MergeONNXModels
from qonnx.core.datatype import DataType
from qonnx.transformation.insert_topk import InsertTopK
from qonnx.transformation.infer_datatypes import InferDataTypes

model = ModelWrapper(build_dir+f"/end2end_{model_name}_tidy.onnx")
global_inp_name = model.graph.input[0].name
ishape = model.get_tensor_shape(global_inp_name)
# preprocessing: torchvision's ToTensor divides uint8 inputs by 255
totensor_pyt = ToTensor()
chkpt_preproc_name = build_dir+f"/end2end_{model_name}_preproc.onnx"
export_qonnx(totensor_pyt, torch.randn(ishape), chkpt_preproc_name)
qonnx_cleanup(chkpt_preproc_name, out_file=chkpt_preproc_name)
pre_model = ModelWrapper(chkpt_preproc_name)
pre_model = pre_model.transform(ConvertQONNXtoFINN())

Now the network needs to be processed further for hardware synthesis, we target the convolutional layers next.
Convolutions in FINN are implemented with convolution lowering, this approach converts convolutional layers into matrix mutiply opeerations, where one of the matrices is generated by a sliding window from the input image.
Two primitives in the FINN RTL library are the Sliding Window Unit (SWU) and Matrix Vector Activation Unit (MVAU), the following operations transform the network into ONNX equivalents of these hardware primitives.

First we perform some small transformations and \verb+Streamline+ the network:
The goal of \verb+Streamline+ is to eliminate floating point operations by moving and collapsing them into one operation.
This becomes clearer when we look at the sub transformations involved in the \verb+Streamline+ transformation.

\begin{jupytercell}

from finn.transformation.streamline import Streamline
showSrc(Streamline)
\end{jupytercell}
\begin{verbatim}
class Streamline(Transformation):
    """Apply the streamlining transform, see arXiv:1709.04060."""

    def apply(self, model):
        streamline_transformations = [
            ConvertSubToAdd(),
            ConvertDivToMul(),
            BatchNormToAffine(),
            ConvertSignToThres(),
            MoveMulPastMaxPool(),
            MoveScalarLinearPastInvariants(),
            AbsorbSignBiasIntoMultiThreshold(),
            MoveAddPastMul(),
            MoveScalarAddPastMatMul(),
            MoveAddPastConv(),
            MoveScalarMulPastMatMul(),
            MoveScalarMulPastConv(),
            MoveAddPastMul(),
            CollapseRepeatedAdd(),
            CollapseRepeatedMul(),
            MoveMulPastMaxPool(),
            AbsorbAddIntoMultiThreshold(),
            FactorOutMulSignMagnitude(),
            AbsorbMulIntoMultiThreshold(),
            Absorb1BitMulIntoMatMul(),
            Absorb1BitMulIntoConv(),
            RoundAndClipThresholds(),
        ]
        for trn in streamline_transformations:
            model = model.transform(trn)
            model = model.transform(RemoveIdentityOps())
            model = model.transform(GiveUniqueNodeNames())
            model = model.transform(GiveReadableTensorNames())
            model = model.transform(InferDataTypes())
        return (model, False)
\end{verbatim}

The next step is to transform the model into FINN-HW layers, the way this is performed in a FINN transformation flow is to first convert layers that can directly be put on the FPGA into their hardware (RTL or HLS) equivalents.
These layers area then separated into a streaming dataflow partition and specialised with parameterized HLS variants.
This results of this transformation is easier to see when visualised:

As seen above, the model has been separated into a dataflow partition and a parent model which contains the dataflow partition.
The dataflow partition contains nodes which are implementable with HLS or RTL, whereas the parent model contains the dataflow partition and any other layers which don't have a hardware equivalent.
In order to create a bitstream to configure the FPGA PL, we operate only on the dataflow partition.

The \verb+InferBinaryMatrixVectorActivation+ transform merges pairs of \verb+XnorPopcount+ layers into a single MVAU and with this we get the added benefit of being able to absorb any following \verb+MultiThreshold+ layers into the MVAU.
The \verb+InferThresholdingLayer+ converts \verb+MultiThreshold+ layers into a single hardware equivalent thresholding HLS layer \verb+Thresholing_rtl+.
The \verb+InferConvInpGen+ transform converts the sliding window Im2Col layers into \verb+ConvolutionInputGenerator+ layers.
The \verb+InferStreamingMaxPool+ transform converts the \verb+MaxPoolNHWC+ to \verb+StreamingMaxPool+ layers.
Finally we perform some tidy up operations which absorb consecutive transposes and useless reshapes between nodes

In an ideal world, an input should go through each layer of the model pipeline with a cycle overhead of 1, however this requires massive hardware overhead and can't fit on hardware resource constrained devices.
A strength of FINN is the ability to fold the model, folding in FINN is the ability to adjust the throughput of certain layers.
A folded design reuses a smaller number of compute units across time, which is slower than the ideal scenario but massively reduces hardware overhead, making these neural networks synthsisable to FPGA.
This is achieved primarily by adjusting Processing Elements (PE) and Single Instruction Multiple Data (SIMD) channels of each matrix-vector layer in a network.

We apply the following folding factors to the model:
\begin{jupytercell}
def fold_cnv(model):
    fc_layers = model.get_nodes_by_op_type("MVAU_hls")
    # each tuple is (PE, SIMD) for a layer
    folding = [
        (16, 3),
        (32, 32),
        (16, 32),
        (16, 32),
        (4, 32),
        (1, 32),
        (1, 4),
        (1, 8),
        (5, 1),
    ]
    for fcl, (pe, simd) in zip(fc_layers, folding):
        fcl_inst = getCustomOp(fcl)
        fcl_inst.set_nodeattr("PE", pe)
        fcl_inst.set_nodeattr("SIMD", simd)
        fcl_inst.set_nodeattr("mem_mode", "internal_decoupled")
        fcl_inst.set_nodeattr("resType", "lut")

    swg_layers = model.get_nodes_by_op_type("ConvolutionInputGenerator_rtl")
    for i in range(len(swg_layers)):
        swg_inst = getCustomOp(swg_layers[i])
        simd = folding[i][1]
        swg_inst.set_nodeattr("SIMD", simd)
        swg_inst.set_nodeattr("ram_style", "distributed")
    return model
model = fold_cnv(model)
model = model.transform(GiveUniqueNodeNames())
model.save(build_dir + f"/end2end_{model_name}_folded.onnx")
\end{jupytercell}

\begin{jupytercell}
from shutil import copy
from distutils.dir_util import copy_tree
from shutil import make_archive

# create directory for deployment files
deployment_dir = make_build_dir(prefix="pynq_deployment_")
model.set_metadata_prop("pynq_deployment_dir", deployment_dir)

# get and copy necessary files
# .bit and .hwh file
bitfile = model.get_metadata_prop("bitfile")
hwh_file = model.get_metadata_prop("hw_handoff")
deploy_files = [bitfile, hwh_file]

for dfile in deploy_files:
    if dfile is not None:
        copy(dfile, deployment_dir)

# driver.py and python libraries
pynq_driver_dir = model.get_metadata_prop("pynq_driver_dir")
copy_tree(pynq_driver_dir, deployment_dir)
make_archive('deploy-on-pynq-cnv-w1a1-minimal', 'zip', deployment_dir)
\end{jupytercell}
